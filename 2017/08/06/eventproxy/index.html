事件机制

2017-08-06
#Js

事件流

可以想象在嵌套着不同半径的同心圆中，当你点击圆心的时候，你实际上点击了所有的圆。所以如果你点击了一个按钮，实际上，你还点击了包含了这个按钮的父元素，这样一层层地触发。一系列的事件触发的顺序就是事件流。
然而，很久很久以前Microsoft和Netscape对于事件触发的顺序有些截然不同的定义。

Event Bubbling(事件冒泡) IE事件流

事件冒泡是指从最具体的元素开始，直到触发至最父层。

Event capturing(事件捕获) Netscape支持的事件机制

事件捕获与事件冒泡相反，事件最开始由真正触发事件的节点的祖先结点开始触发，再一层一层往下到真正的结点。

W3C标准的DOM事件流

分为三个阶段
1.事件捕获阶段： 事件从document一直往下传播到目标元素，依次检查经过的节点是否绑定了该阶段的事件监听函数，如果有，则触发。

2.事件目标阶段：
真正触发事件的元素，并触发该事件函数产生。

3.事件冒泡阶段：
事件从目标元素，一直往上传播到目标元素，依次检查经过的节点是否绑定了该阶段的事件监听函数，如果有，则触发。

与事件绑定的只要函数是

type是事件类型，listener是事件监听函数，而useCapture就是指定该事件是在Capture阶段还是在Bubble阶段触发，默认是false,表明默认的是采取冒泡机制，应该是想兼容IE浏览器。而true代表捕获机制

这样方式绑定的事件机制也是冒泡机制


例如上面的例子,点击粉色区域,即Id为child的时候console结果如下：
1.parent的dom节点 “parent”
2.child的dom节点 “child”
3.#document的dom结点 “undefined”

首先点击的真正元素的是child,浏览器从最上层遍历绑定了捕获阶段事件监听函数的祖先元素，遍历的节点有document和parent，但是只有parent绑定了捕获阶段事件监听函数，所以最先触发的是parent

接着是真正触发的元素child

再接着回溯往上找，遍历的节点还是有parent和doc，但是只有document绑定了冒泡阶段事件监听函数，所以打印了document的dom节点。

基于冒泡机制，所以一般对于子元素数组，一般都采用事件代理的方法去绑定事件，这样可以减少内存空间，而且当子元素动态增减的时候，也不用再次给新增的子元素绑定事件了。

这样，虽然真正点击的元素的li，但是通过冒泡机制，当事件传递到父元素时也会触发。

在IE浏览器中使用attachEvent(type,handler)绑定事件。

如果是取消事件传播，在IE中可以使用window.event.cancelBubble = true
而W3C标准的浏览器可以用e.stopPropagation()
